# 資格登録の抽出精度・導線・100%チェック改善 設計（2026-06-13）

## 背景・課題

UI からの資格登録機能（`/my/certifications/new` + `certificationParser` + `certification-form.ejs`）は既に存在し、学習ガイド URL からドメイン名・割合を自動抽出して手動編集・保存できる。しかし以下のギャップがある:

1. **抽出精度** — gh-500（GitHub Advanced Security）のような新形式で 6 ドメイン中 1 つを取りこぼす。原因は、ドメイン1の見出しだけ範囲区切りに全角チルダ `～`(U+FF5E) を使うが、`parseDomainsFromMarkdown` の区切り文字クラス `[-–—~〜]` が波ダッシュ `〜`(U+301C) しか含まないため。残り5つは ASCII `~` でマッチする。
2. **導線** — グローバルナビ（`views/partials/hud.ejs`）に「マイ資格」へのリンクが無く、登録画面 `/my/certifications/new` に UI から到達できない。
3. **合計100%チェック** — 手動編集時に割合の合計が 100% になっているか UI で検証されない（サーバー側 `normalizeWeightsToSum100` は抽出時のみ動作）。

### gh-500 ガイドの実構造（確認済み）

ドメインは H3 見出しに割合付きで6つ。区切り文字が混在する:

| # | ドメイン | 割合表記 | 区切り文字 |
|---|---|---|---|
| 1 | GitHub のセキュリティ スイート、機能、エコシステムについて説明する | (15～20%) | `～` U+FF5E |
| 2 | シークレット保護の構成と使用 (以前のシークレット スキャン) | (15 ~ 20%) | ASCII `~` |
| 3 | サプライ チェーン セキュリティの構成と使用 (旧称 …) | (15 ~ 20%) | ASCII `~` |
| 4 | コード セキュリティの構成と使用 (以前の …) | (10 ~ 15%) | ASCII `~` |
| 5 | セキュリティ操作: ベスト プラクティス、優先順位付け、修復 | (15 ~ 20%) | ASCII `~` |
| 6 | GitHub セキュリティ スイートの管理 | (10 ~ 15%) | ASCII `~` |

2〜4 は見出し途中にも括弧を持つが、`modernRe` の非貪欲 `(.+?)` が後退して末尾の割合括弧を拾うため、名前に別名括弧を含んだまま正しく抽出できる（回帰テストで固定する）。

## 決定事項

| # | 決定 | 理由 |
|---|------|------|
| D1 | 抽出は既存 regex を surgical に拡張する（アプローチ A）。スキルの概要箇条書き解析の新設（B）や LLM 常用（C）は採らない | 最小・低リスク。LLM フォールバックは温存。C は未検証の新エンドポイント（`models.github.ai`）依存・非決定的でテスト困難 |
| D2 | `parseDomainsFromMarkdown` の `modernRe` の範囲区切り文字クラス `[-–—~〜]` を `[-–—~〜～－−]` に拡張（全角チルダ U+FF5E、全角ハイフン U+FF0D、マイナス U+2212 を追加）。`parseDomainsWithLlm` は LLM プロンプトで区切り文字 regex を持たないため変更不要 | gh-500 のドメイン1取りこぼしを解消。他の全角表記も先回りでカバー |
| D3 | グローバルナビ（`hud.ejs`）に「📚 マイ資格」リンク（→ `/my/certifications`）を追加 | 全ページから登録画面へ到達可能にする |
| D4 | フォームにドメイン割合の**合計%ライブ表示**と「100%に正規化」ボタンを追加（クライアント側で完結） | 手動編集時の 100% 整合を UI で担保。チェックであり保存時の強制ではない |
| D5 | 保存時のサーバー挙動は変更しない | ユーザー要望は「チェック」であり強制正規化ではない。意図的な非100設定を壊さない |
| D6 | 2つ目URL（トレーニング/練習）の抽出利用はスコープ外。既存 `courseUrl` フィールドへの保存のみ | ユーザーがスコープから除外。`courseUrl` は問題生成の追加コンテキストとして既に利用される |

## アーキテクチャ / コンポーネント

既存構成を踏襲し、新規ファイルは作らない。

```
views/partials/hud.ejs        … 「📚 マイ資格」ナビリンク追加（全ページ共通）
  → /my/certifications (既存)  … 「＋ 新規追加」ボタン（既存）
    → /my/certifications/new   … certification-form.ejs

services/certificationParser.js
  parseDomainsFromMarkdown     … modernRe の区切り文字クラス拡張（今回の唯一のロジック修正）
  parseDomainsWithLlm          … 変更なし（プロンプトベースで区切り文字 regex 依存なし）

views/certification-form.ejs
  既存: 抽出ボタン → /my/certifications/extract、ドメイン手動編集、domainsJson hidden
  追加: 合計%ライブ表示 + 「100%に正規化」ボタン（renderDomains/sync に総和計算を追加）
```

## データフロー（変更なしの確認）

1. フォームで study guide URL 入力 → 「URL からドメイン構造を自動抽出」→ `POST /my/certifications/extract`
2. `certificationParser.extractDomains` が MCP fetch → `parseDomainsFromMarkdown`（**今回の修正対象**）→ 失敗時 LLM フォールバック → 合計100に正規化
3. フォームにドメイン配列が描画される → ユーザーが手動編集（**合計%ライブ表示が更新**）
4. 保存 → `POST /my/certifications/new` → `buildCertification` → Cosmos へ（サーバー挙動不変）

## 合計%チェックの UI 仕様

- ドメイン一覧の下に「合計: NN%」を表示。
- `NN === 100` → 緑（`var(--fern)`）。それ以外 → 赤/橙（`var(--crimson)` 系）＋「⚠ 100% になっていません」注記。
- ドメイン0件時は非表示。
- 各 `weight` は空文字/NaN を 0 とみなして合算。
- 「100%に正規化」ボタン: クライアント側で `normalizeWeightsToSum100` と同じロジック（合計0なら均等配分、それ以外は比例配分して余剰/不足を最大ウェイトのドメインで調整）を適用し、`renderDomains` + `sync` を再実行。

## エラーハンドリング / エッジケース

- 区切り文字クラス拡張は既存テスト（AZ-305「から」、SC-400「–」、単一値 `(40%)`、レガシー優先、学習リソース以降の無視）を壊さないこと。
- gh-500 の見出し途中括弧（別名）を含む名前が正しく抽出されること。
- ナビリンク追加は全ビューで `hud.ejs` を include しているため、各ビューが 500 にならないこと（既存の `views.test.mjs` のスモークで担保 + リンク存在アサート追加）。

## テスト方針

- `tests/certificationParser.test.mjs`: gh-500 形式の回帰テストを追加
  - 全角チルダ `～`(U+FF5E) を含む見出しを抽出できる
  - 見出し途中に別名括弧を持つ見出しから末尾の割合を抽出できる
  - 6 ドメイン抽出 & 合計100% 正規化
- `tests/views.test.mjs`: 任意ページのレンダリング結果に「マイ資格」ナビリンク（`/my/certifications`）が含まれることをアサート
- フォームの合計%表示はクライアント JS のためユニットテスト対象外。ビューが 500 にならないこと（既存スモーク）で担保

## スコープ外

- 「スキルの概要」箇条書きの独立パース（アプローチ B）
- LLM 常用化（アプローチ C）
- 2つ目URL（トレーニング/練習）の抽出利用
- 保存時のサーバー側強制正規化
